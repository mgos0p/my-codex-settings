# Career Story

## Overview
未整理の経験メモを、技術面接でそのまま使いやすい説明に構造化する。
事実ベースを守り、本人の役割と貢献を切り分けて整理する。

## Workflow
1. 読み取った情報をプロジェクト単位で分ける。
2. 背景、役割、技術判断、成果の材料を拾う。
3. 面接で説明しづらい曖昧さが残る場合だけ、必要最小限の質問をする。
4. 指定フォーマットに沿って、日本語の箇条書き中心でまとめる。
5. 本人の貢献と、チームやプロジェクト全体の出来事を分けて書く。

## Ask Only When Needed
以下が欠けていて、説明の質が大きく落ちる場合だけ質問する。

- 本人の担当範囲や責任範囲
- 技術選定や実装判断の理由
- 成果や変化の内容
- 学びや再現可能な知見

質問は短く具体的にする。まとめて大量に聞かず、重要なものから聞く。

質問例:
- あなた自身が主担当だった範囲はどこですか。
- その技術判断をした背景や制約は何ですか。
- 結果として何が改善しましたか。数値がなければ定性的な変化でも構いません。

## Output Rules
- 出力は日本語にする。
- 箇条書きを基本にする。
- 誇張しない。
- 書かれていない成果や数値を創作しない。
- 推測で補う場合は、断定せず不確実さが伝わる書き方にする。
- 本人の行動は「自分が何を判断し、何を実装し、何を改善したか」が見えるように書く。
- チーム成果を書く場合でも、本人の貢献を別の箇条書きで明確にする。
- 複数案件が混ざっている場合は、案件ごとに分けて整理する。

## Output Format
以下の見出し順を守る。

### Project Summary
- 1〜2文でプロジェクト概要を書く。

### Situation
- プロジェクトの背景を書く。
- 課題が生まれた理由や制約があれば書く。

### Task
- 本人の役割を書く。
- 任された課題や期待された責任を書く。

### Actions
- 技術的な判断を書く。
- 実装内容を書く。
- 工夫した点やトレードオフへの対応を書く。

### Result
- 結果を書く。
- 成果を書く。
- 学びを書く。

### Technologies
- 使用技術を列挙する。
- 明示されていない技術は安易に補わない。

## Quality Check
出力前に次を確認する。

- Task と Actions に本人の貢献が見えているか。
- Result にプロジェクト全体の成果だけでなく、本人の関与後に起きた変化が入っているか。
- 面接で深掘りされそうな技術判断に理由が添えられているか。
- 断定が強すぎたり、実績を盛って見せる表現になっていないか。
